# Evidence Policy

## Source ranking

Rank sources in this order when possible:

1. Official documentation, standards bodies, direct product pages
2. Release notes, changelogs, repositories, maintainers' direct statements
3. Government records, filings, incident reports, company disclosures
4. Major reporting with clear sourcing
5. Specialist analysis or community posts

Lower-ranked sources can support context, but they should not override stronger primary evidence without explanation.

## Freshness rules

- Prefer the date the event happened over the article publish date.
- Use exact dates when the user asks about “today”, “latest”, “recent”, or similar relative time.
- Check whether a page shows both `published` and `updated` timestamps.
- For software and API questions, verify the current version, release date, or changelog entry before answering.

## Verification depth

- For simple lookups, one strong primary source can be enough.
- For time-sensitive, high-stakes, or comparative tasks, use multiple independent sources.
- When sources conflict, explain the conflict and identify the most authoritative source.

## Answer discipline

- Lead with the answer, then show the strongest evidence.
- Distinguish confirmed fact from inference.
- Surface uncertainty instead of smoothing it over.
- Link the primary sources when the host allows clickable links.

## Common failure modes

- Relying on snippets without opening the page
- Quoting outdated summaries instead of current official docs
- Mixing old and new product versions
- Treating blog reposts as primary evidence
- Using relative dates without resolving them to exact calendar dates
